# OpenJarvis — Full Platform Reference for EVE

> **What this is.** A complete map of the OpenJarvis platform (`jarvis` CLI,
> v1.0.2) that EVE — the `jarvis-sidecar` voice assistant — delegates to. EVE's
> `jarvis_agent` tool runs OpenJarvis under the hood; this document tells the
> voice loop, the agent bridge, and any human or Claude session **what
> OpenJarvis can actually do and how to reach it.**
>
> Generated by live introspection on 2026-06-13 (`jarvis --help`,
> `jarvis registry show …`). When OpenJarvis is upgraded, regenerate with the
> commands in [§12 Keeping this current](#12-keeping-this-current).

> ⚠️ **Two audiences — mind the fold.** This whole file is for **humans and
> Claude sessions**. The **live small voice model** must NEVER be fed this whole
> document — it sees only a distilled hint generated by `openjarvis_capabilities.py`
> (sourced from §1 flow + §10 cheat-sheet). Pasting all 12 sections into a prompt
> burns the context window on channel names and engine ids the model never routes
> to. Human reference here; machine map in the `.py`.

## 1. How EVE relates to OpenJarvis

EVE is the **voice/persona front door**. OpenJarvis is the **agentic backend**.

```
the owner speaks ─► EVE voice loop (bot.py / phone_bot.py)
                    │  has ~23 fast, local voice tools (weather, SMS,
                    │  reminders, calendar, memory, etc.)
                    │
                    └─► jarvis_agent(task)  [agent_bridge.py]
                          tiered chain, order from JARVIS_BRAIN_ORDER
                          (EVE default: local,claude,codex)
                            • local  → jarvis ask --agent orchestrator
                            • claude → claude -p   (Claude Code headless)
                            • codex  → codex exec   (OpenAI Codex CLI)
```

- EVE handles **conversation + her own quick tools** directly.
- Anything heavier — research, files, code, scheduling, multi-step work — goes
  to `jarvis_agent`, which prefers the **local OpenJarvis orchestrator** (free,
  offline, full 38-tool registry) and falls back to cloud brains.
- The orchestrator runs on **the inference box vLLM** (`Qwen3.5-122B-A10B`,
  `http://your-vllm-host:8080`, over Tailscale).

**Server:** OpenJarvis runs as `jarvis-server.service` on port **8000**
(OpenAI-compatible API). `jarvis status` shows daemon health.

## 2. CLI command reference (42 commands)

Grouped by what EVE would use them for. `jarvis <cmd> --help` for live detail.
Commands marked **▶ delegation-relevant** are the ones EVE/the orchestrator
should actually reach for; the rest are operator/maintenance commands.

### 2.1 Ask & chat — the main entry points ▶

| Command | Purpose | Key options |
|---|---|---|
| `jarvis ask QUERY...` | One-shot question to Jarvis. **This is how EVE delegates.** | `-a/--agent` (simple, orchestrator…), `--tools calc,think`, `-m/--model`, `-e/--engine`, `--json`, `--no-stream`, `--no-context` (skip memory), `--research` (hybrid BM25+dense over knowledge.db), `--persona NAME` |
| `jarvis chat` | Interactive multi-turn session. `/quit /clear /model /history`. | `-a/--agent`, `--tools`, `--system`, `--persona` |

**EVE's delegation call today:** `jarvis ask --agent orchestrator --json "<task>"`
(see `agent_bridge.py:_try_local`).

### 2.2 Agents — persistent, scheduled autonomous workers ▶

`jarvis agents …` — create long-lived agents that tick on a schedule, hold
state, bind to channels, and self-improve. Currently **none created**
(`jarvis agents list` → empty); 22 agent *types* are available to instantiate
(see [§4](#4-agent-types-22)).

| Subcommand | Purpose |
|---|---|
| `create -n NAME [-t TEMPLATE] [--type TYPE]` | Create a persistent agent |
| `list` / `status` / `info AGENT_ID` | Inspect agents |
| `ask AGENT_ID MESSAGE` | Immediate response from an agent |
| `instruct AGENT_ID MESSAGE` | Queue an instruction for next tick |
| `start` / `stop AGENT_ID` | Start/stop scheduling |
| `run AGENT_ID` | Run one tick now (testing) |
| `pause` / `resume` / `recover AGENT_ID` | Lifecycle control |
| `bind AGENT_ID --slack/--telegram/--whatsapp` | Bind a channel |
| `channels AGENT_ID` | List channel bindings |
| `logs` / `trace` / `messages` / `search` / `tasks` AGENT_ID | History & traces |
| `learning AGENT_ID [--run]` | Learning history / manual run |
| `errors` | Agents in error / needs-attention |
| `watch [AGENT_ID]` | Live activity feed |
| `templates` | List agent templates |
| `daemon` | Run the agent scheduler standalone |

### 2.3 Scheduler — cron/interval/once tasks ▶

`jarvis scheduler …` — schedule prompts to run automatically.

| Subcommand | Purpose |
|---|---|
| `create PROMPT --type [cron\|interval\|once] --value VAL [--agent A] [--tools …]` | Schedule a task. `--value` = cron expr, seconds, or ISO datetime |
| `list [--status active\|paused\|completed\|cancelled]` | List tasks |
| `pause` / `resume` / `cancel TASK_ID` | Lifecycle |
| `logs TASK_ID [-n N]` | Run logs |
| `run-task AGENT_NAME [--dry-run]` | Run the active task now (launchd-friendly) |
| `start [--poll-interval S]` | Scheduler daemon (foreground) |

**Example EVE could trigger:** `jarvis scheduler create "Summarize unread email
and text me the highlights" --type cron --value "0 6 * * *" --agent orchestrator`

## 7. Memory model — shared vs separate

**Decision: separate by purpose, wiki-canonical, with a one-way bridge.**

| Store | Owner | Holds | Canonical for |
|---|---|---|---|
| **Obsidian wiki** (`…/wiki/jarvis-memory.md`) | EVE (`memory_tool.py`) | Dated fact bullets from `remember`; human-readable, git-tracked | **Durable facts: goals, prefs, commitments, people** |
| OpenJarvis `memory.db` / backends | OpenJarvis | Indexed docs, vectors, agent working memory | Agent task/session retrieval |
| `knowledge.db` | OpenJarvis | Hybrid BM25+dense knowledge store (`--research`) | Research over ingested sources |

- EVE's `remember`/`recall` already write to the **wiki** (verified:
  `JARVIS_MEMORY_PAGE=…/obsidian-vault-second-run/wiki/jarvis-memory.md`).
- **`agent_bridge._build_delegation_context()` feeds task-relevant wiki facts
  into delegated tasks** — same keyword recall the `recall` tool uses, **ranked
  by hit-count, recency tiebreak, hard-capped at `MAX_WIKI_FACTS=5`** — so the
  orchestrator inherits context (the owner's goals, tone) without any DB merging.
  The cap is a named constant in the bridge, not prose.
- Memory backends available: `sqlite`, `dense`, `hybrid`, `knowledge` (5 entries
  in `MemoryRegistry`).

## 8. Identity — shared vs separate

**Decision: the sidecar persona is canonical; OpenJarvis identity is derived.**

| Field | Sidecar (canonical) | OpenJarvis (derived) |
|---|---|---|
| Assistant name | `JARVIS_ASSISTANT_NAME=Eve` (`.env`) → `persona.py` | `SOUL.md` "You are Eve" |
| User name | `JARVIS_USER_NAME=Sam` | `USER.md` "Sam Rivers" |
| Address | `JARVIS_USER_NICK=Sir` | `SOUL.md` 'Address the owner as "Sir"' |
| Aliases | `JARVIS_SELF_NAMES=boss,chief,owner` | — |
| Businesses | `business_context.md` | `USER.md` / `SOUL.md` |

- **Source of truth:** `.env` + `persona.py`. Edit identity there.
- `~/.openjarvis/SOUL.md` + `USER.md` are injected into every delegated task by
  the bridge (so the orchestrator knows who it works for). Keep them in sync;
  `sync_identity.py` can regenerate them from `.env`.
- `agent_bridge._load_identity()` loads `SOUL.md` + `USER.md` as CONTEXT today;
  `_build_delegation_context()` (the new path) wraps it and adds the capability
  map + matched wiki facts. The identity injection is unchanged; the enrichment
  is purely additive and degrades to identity-only if recall fails.

## 10. Delegation cheat-sheet — when EVE should reach for what

EVE keeps her **own fast tools** for: weather, SMS (MacroDroid), reminders,
calendar, email headlines, news, notes, media control, `remember`/`recall`,
opening apps. **Delegate via `jarvis_agent`** for everything below. The bridge
now injects this map so the orchestrator routes correctly.

> 🤖 **This table is the seed of the runtime map.** `openjarvis_capabilities.py`
> distills these rows into a **two-column** form (intent → CLI example), dropping
> the middle "subsystem" column as noise for the model. The three-column version
> below is for humans; the `.py` is the sibling the small model actually sees.
> They are siblings, not duplicates — update both when capabilities change.

| the owner says… | OpenJarvis subsystem | CLI the orchestrator reaches |
|---|---|---|
| "Research X / dig into Y" | deep research | `jarvis ask --research` / `deep_research` agent |
| "Write/edit a file, run this code" | tools | `file_write`, `code_interpreter`, `shell_exec` |
| "Schedule a daily/▶recurring task" | scheduler | `jarvis scheduler create --type cron` |
| "Keep an eye on X / monitor Y" | operators | `jarvis operators activate` / `monitor_operative` |
| "Message me on Telegram/Slack/Signal" | channels | `jarvis channel send --channel-type …` |
| "Index/search my docs" | memory | `jarvis memory index` / `memory search` |
| "Build the morning digest" | digest | `jarvis digest --fresh` / `morning_digest` agent |
| "Look up in my notes/wiki" | knowledge | `jarvis ask --research` over `knowledge.db` |
| "Run a multi-step pipeline" | workflows | `jarvis workflow run` |
| "Connect my Gmail / Obsidian" | connectors | `jarvis connect --path …` |

## 12. Keeping this current

When OpenJarvis is upgraded (`jarvis self-update`), regenerate the live facts.
**Both artifacts must be updated — the doc AND the code map:**

1. Re-run the introspection commands below.
2. Update the counts and tables in this file (§2–§9, §11 snapshot date).
3. **Update `openjarvis_capabilities.py`** — its `_STATIC_FALLBACK` map and the
   intent→CLI rows, if any subsystem/command changed. This is a required step,
   not an afterthought: the live model reads the `.py`, not this doc.
4. Run `python -m pytest tests/test_openjarvis_capabilities.py` to confirm the
   map still asserts (mentions orchestrator, scheduler, workflow, memory, channel).

```bash
jarvis --version
jarvis --help                                  # command list
for g in agents scheduler operators compose workflow memory channel \
         connect model skill eval feedback tool registry config vault; do
  jarvis "$g" --help

jarvis registry show ChannelRegistry           # 30 channels
jarvis registry show EngineRegistry            # 14 engines
```

Then update the counts and tables above, and the verb→subsystem rows in
`openjarvis_capabilities.py` if new subsystems appeared.
